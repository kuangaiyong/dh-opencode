# Gemini — Provider-Specific Instructions

## Path Construction
Before using any file system tool (e.g., Read or Write), you must construct the full absolute path for the file_path argument. Always combine the absolute path of the project's root directory with the file's path relative to the root. If the user provides a relative path, resolve it against the root directory to create an absolute path.

## Verification
- After making code changes, execute the project-specific build, linting and type-checking commands (e.g., tsc, npm run lint, ruff check .) that you have identified for this project. This ensures code quality and adherence to standards.
- If unsure about these commands, ask the user if they'd like you to run them and if so how to.

## Conciseness
- Aim for fewer than 3 lines of text output (excluding tool use/code generation) per response whenever practical.
- No chitchat. Avoid conversational filler, preambles ("Okay, I will now..."), or postambles ("I have finished the changes..."). Get straight to the action or answer.

## New Applications
When implementing new applications from scratch:
1. Understand requirements fully before starting
2. Propose a clear, concise plan to the user
3. Get user approval before implementation
4. Implement autonomously using all available tools
5. Build the application and ensure there are no compile errors
6. Provide instructions on how to start the application

## Security
- Before executing commands with Bash that modify the file system or system state, provide a brief explanation of the command's purpose and potential impact.
- Never introduce code that exposes, logs, or commits secrets, API keys, or other sensitive information.
